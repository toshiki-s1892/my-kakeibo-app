---
name: screen-design-doc
description: Stitchの画面モックアップ（スクリーンショット）から、パーツを採番した画面設計書（docs/design/{screen}.md）を作成・更新する。新しい画面の設計書が必要なとき、または既存画面を確定版で作り直した後の設計書更新に使う。
---

Stitchで作成した画面モックアップを元に、`docs/design/_template.md`に沿った画面設計書を作成する。引数（`ARGUMENTS`）に対象の画面名（例: `categories`、`transactions`）が渡される想定。渡されない場合はユーザーに確認する。

**ファイル名・分割方針（2026-06-22決定）**: 一覧・新規作成・編集・削除のうち2つ以上が存在する画面は`docs/design/{画面名}-list.md`・`{画面名}-create.md`・`{画面名}-edit.md`・`{画面名}-delete.md`に分割する（詳細は[_template.md](../../../docs/design/_template.md)冒頭のコメント参照）。CRUDの概念がない単一画面（home・ai・landing・profile-setup等）は分割せず`{画面名}.md`のままでよい。Dialog/AlertDialogの見た目の共通フレームワークは[modals.md](../../../docs/design/modals.md)を参照し、各CRUDファイルには画面固有のフォーム項目・業務ロジックのみを記載する。

このSkillは[categories-list.mdの試作](../../../docs/design/categories-list.md)で検証済みの手順を一般化したもの。実例として参照してよい（categories関連ファイルは再生成待ちで内容は書き変わる予定だが、フォーマット・節構成の参考としては有効）。

**複数画面をまとめて生成してから最後にこのSkillでまとめて設計書を書く、という進め方はしない**。[`stitch-screen-mockup`スキル](../stitch-screen-mockup/SKILL.md)側で1画面の確定スクリーンショットが`docs/design/screenshots/`に保存できた時点で、次の画面の生成に進む前にこのSkillを呼んで設計書を仕上げる（2026-06-22ユーザー指示）。

## 手順

### 1. 前提確認

- `docs/design/screenshots/{screen}-pc.png`・`{screen}-sp.png`が存在するか確認する
- **Stitch上に画面自体がまだ存在せず、スクリーンショットも撮れない場合は、[`stitch-screen-mockup`スキル](../stitch-screen-mockup/SKILL.md)で機能仕様からモックアップを新規生成してから戻ってくる**（生成・編集はあちらの責務。このSkillは「できあがった画面をスクリーンショット化してdocに落とす」ことに専念する）
- Stitch上に画面は既にあるがローカルにスクリーンショットだけ未保存の場合は、`docs/design/README.md`の「Stitchプロジェクト」セクションのIDを使い、Stitch MCPの`get_screen`でスクリーンショットURLを取得し、`curl`で`docs/design/screenshots/`に保存する
- 対象画面の機能仕様（`docs/specs/features/{feature}.md`）を読み、画面が対応する機能・APIエンドポイント・業務フローを把握する
- **状態パターン（タブ切替・フィルタ適用・単発/定期切替等）のスクリーンショットも対象に含める**。デフォルト状態（PC/SP）だけでなく、`{screen}-{device}-{状態名}.png`として保存済みのパターンがあれば、それらも手順2〜4の対象にしてこのSkillの中で画面設計書に反映する（2026-06-22ユーザー指示）
- このSkill（`screen-design-doc`）の呼び出し自体はドキュメント作成作業であり、`stitch-screen-mockup`側の生成・編集（[承認が必要な操作](../stitch-screen-mockup/SKILL.md#生成編集の実行前に必ず承認を得る)）には当たらないため、**実行前にユーザーの承認を得る必要はない**（2026-06-22ユーザー指示）

### 2. パーツの識別

- PC/SP両方のスクリーンショットをReadツールで開き、画面内の主要パーツをすべて識別する（ヘッダー要素・タイトル・主要コンポーネント・ボタン・ナビゲーション・FAB等）
- 識別の際は仕様外要素も見逃さない。[style-guide.md](../../../docs/design/style-guide.md#現行スクリーンショットと仕様の差分チェック結果再生成前に必読)に既知の傾向（通知アイコン・ヘッダー内ナビリンク等）が載っているので、再発していないか確認する
- 各パーツが機能仕様のどの記述に対応するか（または対応する仕様がない=仕様外）をこの時点で判断しておく
- **PC版とSP版でヘッダー構成等が異なり、パーツの対応が1対1にならない場合がある**（例: PC版はロゴ表示・SP版はロゴなし）。この場合は無理に番号を共通化せず、デバイスごとに別々の番号を振り、パーツ一覧表で「PC#」「SP#」の2列に分けて対応関係を示す
- **状態パターンの番号付け**: デフォルト状態だけでなく、**状態パターン（フィルタ適用中・タブ切替後等）の画像にも必ず番号を振る**（[元画像の削除方針](#元画像の削除番号付け確定後デフォルト)により、番号付け後は元画像を削除するため、番号なしのまま参照され続ける画像を残さない）。デフォルト状態の続きから連番にするか、状態ごとに別の番号体系（例: パターンA-1, A-2…）にするかは、その画面の状態数・パーツの重複度に応じて判断してよい（どちらでも構わない、2026-06-22ユーザー指示）。デフォルト状態と大部分のパーツが共通する場合は連番継続、構成が大きく異なる場合は状態ごとに分けた方が分かりやすい

### 3. Pillowで番号ピンを描画

座標はスクリーンショットを目視して決める。実ピクセルサイズの確認には`python3 -c "from PIL import Image; print(Image.open(path).size)"`を使う（Stitchの画像は表示時の見た目の2倍解像度であることが多い）。

描画は[`annotate_screenshot.py`](./annotate_screenshot.py)（このSkillに同梱、`docs/design/screenshots/`配下への読み書きのみに制限済みで自動承認対象）を使う。

```bash
python3 .claude/skills/screen-design-doc/annotate_screenshot.py \
  docs/design/screenshots/{screen}-pc.png \
  docs/design/screenshots/{screen}-pc-numbered.png \
  '[[x1,y1],[x2,y2],...]'

python3 .claude/skills/screen-design-doc/annotate_screenshot.py \
  docs/design/screenshots/{screen}-sp.png \
  docs/design/screenshots/{screen}-sp-numbered.png \
  '[[x1,y1],...]'
```

ピンは座標リストの順に1番から自動で振られる。スクリプト自体は編集しない（任意のpython実行を自動承認しないため、このスクリプトの呼び出しのみが`.claude/settings.json`で許可されている。座標や画像パスの調整だけで対応できるはずで、スクリプトの拡張が必要になった場合はユーザーに確認する）。

**座標決定→描画→目視確認→調整のループを必ず行う**: 描画後に`Read`ツールで`*-numbered.png`を開き、ピンが意図した要素の近くにあるか確認する。文字に重なって読みにくい場合や、明らかに対象からズレている場合は座標を調整して再描画する（多少テキストに重なる程度は許容してよい。実用上識別できれば十分）。

#### 元画像の削除（番号付け確定後、デフォルト）

`*-numbered.png`の内容に問題がないと確認できたら、その時点で元の番号なし画像（`docs/design/screenshots/{screen}-{device}.png`、および状態パターンの番号なし画像）を削除する。**これは画面ごとに都度行うデフォルトの手順**であり、全画面の作成が完了するのを待つ必要はない（2026-06-22ユーザー指示。ローカルには採番済みスクリーンショットのみを保持する方針、[design-docs-tooling.md](../../../docs/architecture/decisions/design-docs-tooling.md#画面設計書運用stitch)参照）。

削除してもStitch上の元画面（Screen ID）は残るため、後で番号位置を振り直したい場合は`get_screen`でスクリーンショットURLを再取得し、`curl`で保存し直せる。

### 4. `_template.md`を埋める

`docs/design/_template.md`をコピーし、`docs/design/{screen}.md`（CRUDが複数存在する画面は`{screen}-list.md`等、分割方針は[_template.md](../../../docs/design/_template.md)冒頭のコメント参照）として以下を埋める。

- **関連画面**: この画面への遷移元・この画面からの遷移先のみ（全体の遷移図は`screen-flow.md`が別にあるため重複させない）
- **関連API**: 機能仕様の「APIエンドポイント」表から転記する（詳細はリンクで参照させ、ここでは対応関係のみ）
- **採番済みスクリーンショット**: `*-numbered.png`を埋め込み、Stitch Screen IDを記載。**状態パターンがある場合は、デフォルト状態と並べて各パターンの採番済み画像・Screen IDも記載する**（節を分けるか表に行を追加するかは画面に応じて判断）
- **パーツ一覧**: PC/SPで番号が共通の場合は1列、異なる場合は2列（手順2参照）。仕様外のパーツも記載し、その旨を一言添える
- **状態一覧**: 空状態・エラー状態・ローディング状態に加え、**タブ切替・フィルタ適用等の状態パターンも記載する**（パターンごとに何が変わるか、対応するスクリーンショットへのリンクを書く）。モックアップに表現がない状態は「モックアップ上の表現はなし」と明記し、想定される挙動を記述する。デフォルトデータが常に存在するなど「空状態が発生しない」場合はその理由を書く
- **レスポンシブ差分**: 単純なレイアウト幅の違いは書かない。挙動・構成要素自体が異なる場合のみ記載する
- **採用した方向性**: 各要素が機能仕様のどの記述に対応するかを`[xxx.md](../specs/features/xxx.md#見出し)`形式でリンクしながら書く
- **既存実装との差分**: 未実装なら「未実装のため差分なし。」
- **仕様外要素**: パーツ番号と対応させた表形式にする（番号→内容→対応方針）。デフォルトのテキスト記述でも良いが、表の方が後から見返しやすい
- **更新履歴**: 作成日と変更内容

### 5. 横展開の更新

- 新たに仕様外要素・差分を発見した場合は、[style-guide.mdの差分チェック結果](../../../docs/design/style-guide.md#現行スクリーンショットと仕様の差分チェック結果再生成前に必読)の表に追記する（Stitch確定版再生成時に同じ問題を再発させないため）
- [docs/design/README.mdの画面一覧](../../../docs/design/README.md#画面一覧)の該当行を更新する（採番済みスクリーンショットへのリンク、状態列に「新テンプレートで試作済み」等を反映）

## 注意点

- スクリーンショットの取得・Pillowでの画像生成は実装ファイルではないため、CLAUDE.mdの「明示的に依頼されない限りファイルの変更・作成を行わない」の対象外として進めてよい（画面設計書自体がユーザーの依頼対象）
- `docs/specs/features/{feature}.md`に書かれていない要素は仕様外として扱い、勝手に「仕様のはず」と判断しない。判断に迷う場合はユーザーに確認する
