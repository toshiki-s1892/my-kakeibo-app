# Stitchプロンプト・スタイルガイド

Stitchで画面を生成・再生成する際に**毎回のプロンプトへ要約して含めるべき**ルール集。Stitchはデザインシステム（`assets/4009687132338552144`）のテキストや過去の指示を生成のたびに自動再計算・忘却することがあるため、重要な制約はデザインシステムに委ねず、都度のプロンプト本文に直接書くこと（[Stitch運用上の既知の制約](#stitch運用上の既知の制約次回大規模生成時の参考)も参照）。

このファイルは「確定版」再生成（ユーザーのgoサイン後）の直前に必ず読み返し、プロンプトに反映する。

## ブランド

- アプリ名は**「かけぼ」のみ**。`FamMoney`・`Bloom Budget`・`Kakebo Modern`等の別名・サブブランド名を生成しない
- **ロゴは実装済みの[`apps/web/components/Logo.tsx`](../../apps/web/components/Logo.tsx)と完全に一致させる**。新しいロゴ画像・別のアイコン・テキストのみのワードマーク等を独自に生成しない。具体的な見た目:
  - 56×56px（`w-14 h-14`）、角丸16px相当（`rounded-2xl`）、軽い影（`shadow-md`）の正方形コンテナ
  - コンテナの背景色は`#10b981`（プライマリグリーン、`bg-primary-100`）
  - コンテナの中央に、白色（`#FFFFFF`）のGoogle Material Symbols「savings」アイコン（貯金箱のシルエット）を配置（[logo.svg](../../apps/web/public/icon/logo.svg)参照）
  - ロゴの右側にアプリ名「かけぼ」のテキストを並べる（ヘッダー等、ロゴ+テキストの組み合わせが必要な箇所）
  - **プロンプトには毎回上記の見た目を具体的に記述する**。「ロゴ」と一言だけ指示すると画面ごとに異なる画像が生成されるため、「56x56pxの角丸正方形、背景#10b981、中央に白色の貯金箱シルエットアイコン」のように再現可能な記述にすること
- フッター・利用規約・プライバシーポリシー等のページは本プロジェクトに存在しない。生成画面にそれらのリンクを含めない

## カラーパレット

実装側のTailwindトークン（`packages/tailwind-config/shared-styles.css`の`--color-primary-100: #10b981`等）に近い色味として、Stitchデザインシステムでは以下を採用している。ピクセル単位での一致は不要だが、系統色は厳守すること。

| 用途 | 色味 | 参考hex |
|---|---|---|
| プライマリ（メインボタン・アクティブ状態・ブランドカラー） | エメラルドグリーン | `#006c49` 〜 `#06b77f` |
| 背景 | ごく薄いミントグリーン | `#f4fbf4` |
| カード背景 | 白 | `#ffffff` |
| 支出 | ピンク/ローズ系 | カードやアイコン背景に使用 |
| 収入 | グリーン系（プライマリと区別できる明るめの緑） | |
| 収支・バランス | ブルー系 | プライマリの緑と被らない色にする |
| AIアドバイス・注意喚起バナー | アンバー系背景 | 警告アイコンにも使用 |
| カテゴリアイコン・トピックカードの多色使い | ブルー・アンバー・ピンク等を組み合わせる | ミントグリーン一色に寄せない |
| エラー・削除確認ボタン | 赤 | メインアクセントとしては使わない（削除確認ダイアログの「削除する」ボタンのみ） |

**禁止**: コーラル/レッド系を画面のメインアクセントカラーとして使う（本格的アドバイス画面で過去に発生）。紫・原色に近いビビッドカラーの新規導入。

**2026-06-22対応**: デザインシステム自体（`assets/4009687132338552144`）のtertiaryカラーが、styleGuidelines本文に「Tertiary Coral」として明記されており、コーラル系混入の根本原因になっていたため、デザインシステムの`overrideTertiaryColor`をブルー系（`#3b5bdb`）に修正済み（プロンプト側の都度の指示だけでなく、デザインシステム自体を修正した）。ただし**紫の混入は別途カラーカタログ生成時に発生**しており（後述の差分チェック結果参照）、デザインシステムの修正だけでは紫の混入を防げない。プロンプトでの明示的な禁止も併用すること。

## 角丸・形状

大きめの角丸（`ROUND_EIGHT`相当、カードは1rem前後）で統一する。アイコンは「線の先端が丸い、ぽってりとした柔らかいスタイル」を指示するが、**Stitchはアイコンの描画スタイル自体の変更指示が反映されないことが多い**（タイトル・ラベルの文言は反映されるがアイコン描画は変化しない傾向を確認済み）。指示はして良いが、反映されなくても再生成を粘らないこと。

**主要CTAボタン（送信・保存・登録・追加等）の角丸（2026-06-27決定）**: pill形状（`rounded-full`）ではなく、入力欄と同じ中程度の角丸（`rounded-md`〜`rounded-lg`程度）に統一する。pillはFAB・下部ナビのアクティブ項目・メンバーチップ専用とする。`modal-self-edit`の「保存する」・`transactions-form-recurring`の「登録する」は中程度の角丸だが、`profile-setup`の「はじめる」・`modal-family-add`の「追加する」・`landing`の「無料で始める」がpillになっており、画面間で一貫していなかった（Stitch生成ブレ）。日付選択等のフォーム部品トリガー（例: `BirthdayPicker`）も同様に入力欄（`rounded-md`）に揃える。

## 共通レイアウト

- **ヘッダー**: **画面タイトル + Clerkの`UserButton`（アバター）のみ**（[architecture/overview.md](../architecture/overview.md#app-レイアウト構成)の確定方針）。**通知アイコン（🔔）は仕様に存在しないため含めない**。本アプリには通知機能自体が存在しない（プッシュ通知・お知らせ一覧等の機能仕様がどこにもない）にもかかわらず、現行の確定済み画面は[共通パーツのコンポーネントカタログ](./common-components.md)が通知アイコンを含めて生成され、それを参照する形で他の全画面に伝播してしまった。再生成時は**共通パーツ（コンポーネントカタログ）を最初に直し、そこから個別画面を生成する順序**を必ず守ること（[README.mdの作成順](./README.md#次にやること)参照）
- **下部固定ナビゲーション**: 5項目固定、日本語ラベルで統一: 「ホーム」「取引記録」「カテゴリ管理」「家族構成管理」「本格的アドバイス」。英語ラベル（Home/Transactions等）を使わない。最大幅1200px・中央寄せ。アクティブ項目はピル型の緑背景+ラベル表示、非アクティブはアイコンのみ
- **FAB**: 「+ 取引を追加」のみ。複数のFABを配置しない。タップ時のポップアップ2択は「手入力で作成」「レシートから作成」（語尾を「〜で作成」に統一、2026-06-22変更）
- **データテーブル行**: チェックボックス（左端）+ 円形カラーアイコン + メインテキスト/サブテキストの2行表示 + 右端の数値（プラス＝緑、マイナス＝赤）。取引記録・カテゴリ管理・家族構成管理など一覧画面はすべてこの構成に統一する（[コンポーネントカタログ](./common-components.md#採用した方向性)で定義）
- **タブ・セグメントコントロール**: 下線型（アクティブ項目の下に太い緑の下線、非アクティブはグレーテキスト）に統一する。ボタン型・ピル型など画面によって見た目が変わらないようにする。取引記録の「一覧/定期取引」タブで、ボタン型とチェックボックス無しテーブルの組み合わせがランダムに生成され、デフォルト状態（下線タブ+チェックボックス付きテーブル）と見た目が別物になった事例があるため特に注意（後述の差分チェック結果参照）
- **メンバーチップ**: 「本人」「パパ」「ママ」「子ども」のラベルチップ。選択中はプライマリグリーンの塗り、非選択中はグレーアウトラインのピル形状
- **フォーム入力欄の背景色**: Input・Select等の入力欄の背景は白で統一する。ページ背景色（薄緑）を引き込まない（2026-06-26決定。profile-setupの入力欄がページ背景同系の薄緑に着色されていたが、意図した差別化ではなくStitchの生成ブレと判断し、白背景に統一することにした）
- **性別3択ボタン**: 「男性/女性/その他」の3択ボタン。メンバーチップと同様に、選択中はプライマリグリーンの塗り+白文字、非選択中はテキストのみ（枠・背景は白）に統一する（2026-06-26決定。基準は[family-members-edit.md](./family-members-edit.md#採用した方向性)の本人(SELF)編集Dialog。profile-setupは薄緑の塗り、family-members-createはセグメントトラック+白カプセルと、画面によって見た目が異なっていたため統一した）

## 仕様との整合（過去に発生した仕様超え・抜け漏れ）

- **`regionCode`（居住地域）は都道府県のみ**（1〜47の単一コード、[overview.md](../specs/overview.md#都道府県コードregion_code)参照）。**市区町村のプルダウンを追加しない**（プロフィール設定・本人編集モーダルで過去に誤って2段階選択を指示してしまった）
- **世帯合計・世帯全体の集計は表示しない**。ホーム画面は家族メンバーごとのスライド切り替えのみ（[dashboard.md](../specs/features/dashboard.md#概要)参照）
- **AIアドバイス・簡易分析バナーは指示された画面（ホームの簡易分析・本格的アドバイス画面）以外に追加しない**。カテゴリ管理・家族構成管理・取引記録に紛れ込む事例が複数回発生した
- カテゴリに「メモ・説明文」フィールドは存在しない（DBスキーマに列がない）。表示しない

### カテゴリ管理で確認済みの未表現（2026-06-22ユーザー指摘）

旧版スクリーンショットのレビューで、仕様には存在するがモックアップ上で表現されていなかった項目。**カテゴリ管理の再生成時は必ず含めること**。

- **子カテゴリの編集導線**: [親カテゴリ](../specs/features/categories.md#親カテゴリグラフ上のグルーピング)の仕様上、子カテゴリも自分のものであれば編集・ピン留め可能。一覧上で子カテゴリ行にも編集アイコンが表示されることを明示的に確認できるようにする
- **自分が追加した独自カテゴリの表示**: 旧版はデフォルトカテゴリのみが並んでおり、ユーザー自身が追加したカテゴリの行の例がなかった。最低1件は独自カテゴリの行を含める
- **アイコン・色選択フォーム**: [カテゴリアイコン・背景色](../specs/features/categories.md#カテゴリアイコン背景色)の仕様（キュレーションされたアイコン一覧から選ぶピッカーUI、8〜10色のパステルスウォッチ）を、新規追加・編集Dialog（[modals.md](./modals.md)）で必ず表現する
- **ピン留めON状態の表記**: ピンアイコンは「線のみ（OFF）/塗り（ON）」の2状態を持つ仕様だが、旧版は全行が同じ見た目に見えた。ON状態の行を最低1件含める

## 画面パターン・状態デザインの方針（2026-06-22決定）

旧版レビューで「タブ切替後の状態」「ラジオボタン切替時の表示変化」「フィルター適用後」「エラー状態」「空状態」のデザインがどの画面にも存在しないことが判明した。再生成時は以下の方針で対応する。

- **タブ・ラジオボタン切替で表示が変わるケース**（例: 取引記録の「一覧/定期取引」タブ、取引登録フォームの「単発/定期」切替）: 各状態を**別のStitchスクリーンとして生成する**。デフォルト状態のスクリーンと同一のヘッダー・ナビ・配色を維持し、切り替えた状態のコンテンツのみ変更する
- **フィルター適用後の状態**（例: 取引記録の期間・カテゴリ・家族メンバー絞り込み）: 同様に別スクリーンとして生成し、適用中フィルターのチップ表示・件数表示も含める
- **生成方法（2026-06-22変更）**: 上記の別スクリーンは、`generate_screen_from_text`で画面全体をテキストから再構築するのではなく、**`generate_variants`で確定済みの基準スクリーン（`selectedScreenIds`）を渡し、`variantOptions.creativeRange: "REFINE"`、`aspects: ["TEXT_CONTENT"]`（列構成自体が変わる場合は`"LAYOUT"`も追加）を指定して生成する**。`generate_screen_from_text`はテーブル・タブの見た目をゼロから再解釈するため、同一画面内のタブ間でテーブル行・タブのスタイルが別物になる不整合が発生した（取引記録の「一覧」と「定期取引」タブで実例あり。下記差分チェック結果参照）。`generate_variants`は基準スクリーンの構造を引き継ぐため、この不整合を防げる見込み
- **エラー状態・空状態**: 画面ごとに個別生成するとスクリーン数が大幅に増えるため、**汎用パターンはコンポーネントカタログ側にまとめ、個別画面では基本的に生成しない**（テキストでの説明に留める）。ただし、その画面固有の文言・CTAが重要な場合（例: ホーム画面の「この月の支出はまだありません」）は、メンバー切替等の別の状態パターンと合わせて1枚にまとめるなど、生成コストを抑えつつ表現する
- 状態パターンのスクリーンショットも`docs/design/screenshots/`にローカル保存する（ファイル名は`{screen}-{device}-{状態名}.png`、例: `home-pc-child-empty-state.png`、`transactions-pc-recurring-tab.png`）。各画面のmdファイルの該当節（採用した方向性・状態一覧等）からStitch Screen IDとあわせて参照する

確定版再生成前に、現行の全確定スクリーンショット（`docs/design/screenshots/`、再生成前の現行版）を機能仕様と突き合わせて確認した結果。**再生成時はこれらを再発させないこと**。

| 取引登録フォーム（定期モード、PC・1回目の生成） | ヘッダーに通知アイコン（ベルマーク）が再発した（共通パーツでは既に除去済みのはずだった） | 通知アイコン除去の指示は毎回フルで含める必要がある（1〜2点しか直さない傾向と同様、除去済みの制約も再度書き漏らすと再発する）。「ヘッダーはタイトル+アバターの2要素のみ」と明示して再生成した |
| 取引記録（フィルタ適用状態） | FABが指示した横長ピル型「+ 取引を追加」ではなく、円形の+ボタンのみで生成された | 軽微な差分として許容し、画面設計書の仕様外要素に記録。実装時はコンポーネントカタログの確定FAB（横長ピル型）を使う |
| 取引記録（「一覧」タブ/「定期取引」タブ、PC） | 同一画面内のタブなのに、テーブル行（チェックボックス・円形カラーアイコンの有無）とタブの見た目（下線型/ボタン型）が別物になっていた。定期取引タブのFABには絵文字フォントのフォールバック文字「add」が紛れ込んでいた | コンポーネントカタログにテーブル行・タブ・メンバーチップの定義を追加した上で、`generate_screen_from_text`での個別再構築ではなく`generate_variants`（[画面パターン・状態デザインの方針](#画面パターン状態デザインの方針2026-06-22決定)参照）で基準スクリーンから生成し直す |

上記の通り、**通知アイコン・古いロゴ・フッターリンクは「共通パーツ（コンポーネントカタログ）」由来の問題が個別画面に伝播したものが多い**。これが[次にやること](./README.md#次にやること)で「カラーカタログ→コンポーネントカタログ→共通パーツ→個別画面」の順で直す方針を取る理由。

## モーダル・ダイアログの構成パターン

[modals.md](./modals.md)で確定したパターンを踏襲する。

- **Dialog（フォーム入力）**: タイトル+右上×アイコン、フォーム本体、フッターに「キャンセル」（グレーテキストボタン）+ プライマリアクション（エメラルドグリーン塗りボタン）を右寄せ
- **AlertDialog（削除確認）**: アンバー警告アイコン、タイトルに対象名を含める、本文に削除の影響範囲を具体的に記載、フッターは「キャンセル」+ 赤系「削除する」を右寄せ

## Stitch運用上の既知の制約（次回大規模生成時の参考）

Stitch MCPツール自体の挙動に関する注意点（プロンプトの内容とは別軸）。

- **`generate_screen_from_text`はタイムアウトしても裏側で生成が継続していることが多い**。タイムアウト後に同条件で即再試行すると重複生成の原因になるため、まず`list_screens`で新しいタイトルが現れていないか確認すること
- **`list_screens`はプロジェクト内の画面数が増えると全件を返さない**（観測上13〜30件程度で打ち切られ、どの範囲が返るかは呼び出しごとに変動する）。新規生成した画面のIDが分かっている場合は`get_screen`で直接確認する方が確実
- **デザインシステムの`roundness`等は生成・編集のたびにStitch側が自動再計算して上書きすることがある**。重要な指示はデザインシステムのテキストに頼らず、都度のプロンプトに直接書くこと
- **Stitchは仕様にない要素を独自に追加してくることがある**（AIアドバイスバナーの誤挿入、ヘッダー内の重複ナビリンク、存在しないプライバシーポリシーへのリンク、仕様を超えた入力項目の追加等）。生成後は必ずスクリーンショットを確認し、各画面ファイルの「仕様外要素」節に記録すること
- **Stitchは一度の指示で全制約を守らないことが多く、1〜2点だけ直して別の問題が再発する傾向がある**。再生成時は前回の指摘を省略せず、フルの制約リストを毎回プロンプトに含めること
- **`edit_screens`は成功レスポンス（テキスト・dom_operations）を返しても、実際にはファイルが更新されないことがある**（2026-06-22に2回連続で確認）。`edit_screens`実行後は必ず`get_screen`を呼び直し、`htmlCode.name`・`screenshot.name`のファイルIDが実行前と変わっているか確認すること。**変わっていなければ、`edit_screens`をリトライするのではなく`generate_screen_from_text`で一から再生成する**方が確実
- **PC画面で、確定済みの下部固定ナビゲーションの代わりに左サイドバーナビゲーション（縦並びメニュー+プロフィールカード）が生成されることがある**（取引記録の定期取引タブで発生）。本プロジェクトはPC専用のSidebarコンポーネントを採用しない方針（[architecture/overview.md](../architecture/overview.md#app-レイアウト構成)）のため、プロンプトに「左サイドバーは絶対に使用しない、ナビゲーションは下部固定バーのみ」を明示し、生成後のスクリーンショットで毎回確認すること
- **`generate_screen_from_text`は同一画面の別状態（タブ切替等）を都度ゼロから文章で再構築するため、テーブル・タブ等の共通コンポーネントの見た目が画面ごとにブレる**。状態パターンの生成は`generate_variants`（基準スクリーンを`selectedScreenIds`に渡し、`creativeRange: "REFINE"`、`aspects: ["TEXT_CONTENT"]`中心）を優先して使うこと（[画面パターン・状態デザインの方針](#画面パターン状態デザインの方針2026-06-22決定)参照）
- **`screenshot.downloadUrl`（`lh3.googleusercontent.com/aida/...`）は何も付けずに`curl`すると低解像度のサムネイル（長辺512px程度）しか返らない**。URL末尾に`=s0`を付けると元解像度（実際の画面サイズ、例: 2560×4264）が取得できる。スクリーンショットを保存・Pillowで採番する際は必ず`=s0`を付けてダウンロードすること（2026-06-22に判明。これより前に保存した一部のスクリーンショットが低解像度のままになっている可能性があるため、採番作業時に解像度を確認し、低い場合は`=s0`付きで再取得する）
- **タイムアウト後に短い間隔で何度もリトライすると、各回が裏側で別々に成功し、同じ依頼内容の画面が3〜6件単位で重複生成される**（2026-06-23、サインイン/サインアップ・一括削除確認ダイアログの生成で繰り返し観測。`list_screens`の件数が1回の待機サイクルで一気に3〜6件増えることがあった）。1回タイムアウトしたら即座に同条件で再投入せず、**1〜2分待って`list_screens`で新規タイトルの有無を確認するサイクルを2〜3回試した上で、それでも現れない場合のみ次のリトライに進む**。重複が発生した場合は、採用しなかった候補のIDを[README.mdのStitch管理画面での削除候補](./README.md#stitch管理画面での削除候補任意手動作業)に追記する

## プロンプトを書くときのチェックリスト

再生成・新規生成のプロンプトを書く前に、以下を確認する。

1. アプリ名「かけぼ」、ロゴは[`Logo.tsx`](#ブランド)と一致する具体的な見た目（56x56px角丸正方形・背景#10b981・白の貯金箱アイコン）で明記したか。別デザインのロゴ画像を生成させていないか
2. **ヘッダーに通知アイコンを含めていないか**（画面タイトル+UserButtonのみ）。フッター・存在しないページへのリンクを含めていないか
3. 配色（プライマリ・支出/収入/収支の色分け）を上記パレットに沿って具体的に指示したか
4. ヘッダー・下部ナビ（日本語ラベル5項目）・FAB・**テーブル行・タブ・メンバーチップ**の構成を「[共通パーツ](./common-components.md)」の確定内容と一致させたか
5. このプロンプトで対象外の機能（AIバナー・世帯合計・存在しないページへのリンク等）を「追加しない」と明示したか
6. 対象画面の[機能仕様](../specs/)を読み直し、フィールドの粒度（例: regionCodeは都道府県のみで市区町村は無し）が一致しているか確認したか
7. [現行スクリーンショットと仕様の差分チェック結果](#現行スクリーンショットと仕様の差分チェック結果再生成前に必読)に記載した過去の差分を再発させない指示を含めたか
8. 前回の生成で指摘した「仕様外要素」を再度明示的に禁止リストへ含めたか（Stitchは1〜2点しか直さない傾向があるため、毎回フルの制約リストを書く）
9. **「左サイドバーは使用しない、ナビゲーションは下部固定バーのみ」を明示したか**（PC画面で特に発生しやすい）
10. タブ切替・ラジオボタン切替・フィルター適用など、[画面パターン・状態デザインの方針](#画面パターン状態デザインの方針2026-06-22決定)に該当する状態がある画面か確認し、必要なら別スクリーンとして生成する計画に含めたか
11. `edit_screens`で修正する場合、実行後に`get_screen`でファイルIDの変化を確認する手順を踏むか（変化がなければ再生成に切り替える）
